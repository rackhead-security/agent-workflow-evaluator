## 1. Purpose

Agent Workflow Evaluator is a deterministic, local-first static analysis tool for reviewing the declared security properties of AI-agent workflows before deployment.

The evaluator accepts a machine-readable workflow description, validates it against a versioned schema, evaluates explicit data-flow paths and control settings, and emits evidence-backed findings in Markdown and JSON.

The evaluator does not execute agents, prompts, tools, or external requests. It does not prove that a system is secure. Its role is to make design-time risks visible, reviewable, and testable.

## 2. v0.1 goals

The first release must:

1. Load a YAML workflow document into typed models.
2. Reject invalid or internally inconsistent documents with actionable errors.
3. Model explicit flows between data sources, tools, memory, and outputs.
4. Run eight deterministic security rules.
5. Produce Markdown and JSON reports.
6. Provide a command-line interface suitable for local use and CI.
7. Include safe synthetic fixtures and automated tests.
8. Run without network access or an LLM API.

## 3. Non-goals

Version `v0.1.0` will not:

- execute prompts, tools, agents, code, or workflows;
- connect to production systems;
- scan source code or runtime traces;
- inspect credentials or secrets;
- perform dynamic red teaming;
- use an LLM to classify risk;
- claim formal verification or complete security coverage;
- replace human architecture review;
- automatically remediate production configurations;
- model every property of an agent framework.

## 4. Design principles

### 4.1 Deterministic

The same workflow, policy, rule set, and evaluator version must produce the same findings.

### 4.2 Evidence-first

Every finding must identify the exact object or flow that triggered it and include a machine-readable evidence path.

### 4.3 Local-first

Core evaluation must run offline. No telemetry, remote API, or external model dependency is required.

### 4.4 Safe by construction

Fixtures use synthetic data and invalid domains. The project must not contain credentials, real exploit chains, malware, stealth tooling, or unauthorized-access instructions.

### 4.5 Small and reviewable

The initial schema and rule set must remain understandable enough for a reviewer to inspect without specialized infrastructure.

### 4.6 Honest limitations

Reports must state that static declarations may be incomplete or inaccurate and that passing the evaluator does not prove runtime security.

## 5. Supported input

The primary input is a UTF-8 YAML file.

```bash
rackhead-eval workflow.yaml
```

The root document must contain:

```yaml
schema_version: "0.1"
workflow:
  id: demo-support-agent
  name: Demo Support Agent
  version: "0.1.0"
  owners:
    - rackhead-security-demo

risk_profile:
  deployment_stage: internal
  max_autonomy: approved_actions

data_sources: []
tools: []
memory: null
outputs: []
flows: []
approvals: []
monitoring: {}
exceptions: []
```

Unknown fields are rejected by default in `v0.1` to prevent silent misspellings. A future release may add a compatibility mode.

## 6. Core data model

### 6.1 Workflow metadata

```yaml
workflow:
  id: string
  name: string
  version: string
  description: string | null
  owners:
    - string
```

Requirements:

- `id` must be unique within the document.
- `id` values use lowercase letters, digits, hyphens, and underscores.
- `owners` must contain at least one entry.
- `version` is an opaque workflow version string in `v0.1`.

### 6.2 Risk profile

```yaml
risk_profile:
  deployment_stage: toy | internal | production
  max_autonomy: draft_only | read_only | approved_actions | autonomous_actions
```

These values influence rule severity but do not suppress findings.

### 6.3 Data sources

```yaml
data_sources:
  - id: support_tickets
    name: Support tickets
    type: document_store | web | email | chat | database | file_system | api | synthetic
    trust: trusted | mixed | untrusted
    sensitivity: public | internal | confidential | restricted
    access_control: none | user_scoped | service_scoped | repository_scoped
```

Definitions:

- `trust` describes whether content may contain attacker-controlled or unreviewed instructions.
- `sensitivity` describes the highest expected data classification.
- `access_control` describes the narrowest enforced retrieval boundary declared by the workflow owner.

### 6.4 Tools

```yaml
tools:
  - id: send_customer_email
    name: Send customer email
    mode: read | internal_write | external_write | destructive | code_execution
    identity: user_delegated | service_account | anonymous | unknown
    requires_approval: true
    allowed_domains:
      - example.invalid
    allowed_paths: []
    logs_arguments: true
    logs_results: false
```

Requirements:

- `allowed_domains` and `allowed_paths` are explicit lists.
- An empty list means no declared allowlist, not unrestricted safety.
- `requires_approval` describes approval before execution of this tool action.
- Tool identity is evaluated separately from approval.

### 6.5 Memory

```yaml
memory:
  id: agent_memory
  enabled: true
  writes_allowed: true
  review_required: true
  retention:
    mode: none | session | fixed_days | indefinite
    days: 30
  sensitivity: confidential
```

Rules:

- `days` is required only when `mode: fixed_days`.
- `days` must be a positive integer.
- `memory` may be `null` when disabled or absent.
- When `enabled: false`, writes must also be false.

### 6.6 Outputs

```yaml
outputs:
  - id: customer_email
    name: Customer email
    type: chat | file | email | ticket | api | public_web | code_change
    boundary: internal | external | public
    sensitivity_allowed: public
```

`boundary` is used for egress analysis. `sensitivity_allowed` declares the highest intended classification for the output.

### 6.7 Explicit flows

The evaluator must not infer data movement solely from the existence of sources, tools, or outputs. Every analyzed path is represented explicitly.

```yaml
flows:
  - id: ticket_to_customer_email
    from: support_tickets
    through:
      - search_tickets
      - send_customer_email
    to: customer_email
    carries:
      - confidential
    transforms:
      - summarize
    requires_approval: true
```

Requirements:

- `from` references a declared data source, memory object, or tool.
- Every `through` entry references a declared tool.
- `to` references a declared output, memory object, or tool.
- References must exist.
- A flow cannot contain a repeated node in `v0.1`.
- `carries` declares possible data classifications.
- `transforms` is descriptive and does not automatically sanitize data.
- `requires_approval` is a flow-level assertion and does not override a tool configured with `requires_approval: false`.

This explicit graph enables precise findings such as restricted data reaching a public output.

### 6.8 Approvals

```yaml
approvals:
  - id: approve_external_email
    actions:
      - tool:send_customer_email
      - flow:ticket_to_customer_email
    approver_role: support_lead
    required: true
```

Requirements:

- Action references use `tool:<id>` or `flow:<id>`.
- Referenced objects must exist.
- A declared approval counts only when `required: true`.
- The evaluator does not verify that approval is enforced at runtime.

### 6.9 Monitoring

```yaml
monitoring:
  log_tool_calls: true
  log_tool_arguments: true
  log_tool_results: false
  log_retrieved_sources: true
  alert_on_external_write: true
  alert_on_policy_block: true
  retention_days: 90
```

`retention_days` must be a positive integer when present.

### 6.10 Exceptions

Exceptions permit documented acceptance of a specific finding without deleting it from the report.

```yaml
exceptions:
  - rule_id: RH-AWE-006
    object_ref: tool:local_python_sandbox
    justification: Toy-only local execution with no network access
    owner: security-reviewer
    expires_on: 2026-10-01
```

Rules:

- Exceptions require a non-empty justification and owner.
- `expires_on` is required.
- Expired exceptions do not suppress status.
- Exceptions change a finding state to `accepted_exception`; they do not remove the finding.
- Exceptions cannot change schema validation errors.

## 7. Schema validation

Validation has two stages.

### 7.1 Structural validation

Examples:

- invalid enum;
- missing required field;
- incorrect type;
- unknown field;
- invalid identifier;
- invalid retention configuration.

### 7.2 Referential validation

Examples:

- flow references a missing source, tool, output, or memory object;
- approval references a missing tool or flow;
- duplicate IDs;
- repeated nodes in a flow;
- disabled memory with writes enabled;
- exception references an unknown rule or object.

Validation errors must include:

- error code;
- human-readable message;
- YAML path where possible;
- referenced value;
- suggested correction where practical.

Example:

```text
RH-SCHEMA-REF-001
flows[0].through[1]: unknown tool id "send_email"
```

Schema errors stop rule evaluation and return exit code `2`.

## 8. Rule catalog

Rule IDs are stable public identifiers. Internal function names may differ, but reports, policies, fixtures, and documentation use only the `RH-AWE-###` form.

### RH-AWE-001 — External write without approval

**Default severity:** high

Trigger when:

- a tool has `mode: external_write`;
- and the tool does not require approval;
- and no required approval references that tool.

Evidence must identify the tool and approval configuration.

### RH-AWE-002 — Destructive action without approval

**Default severity:** critical in production, high otherwise

Trigger when:

- a tool has `mode: destructive`;
- and no required approval protects the tool.

### RH-AWE-003 — Untrusted input reaches an external or public output

**Default severity:** high

Trigger when:

- a flow originates from a source with `trust: mixed` or `trust: untrusted`;
- and terminates at an output with `boundary: external` or `boundary: public`.

A transform such as `summarize` does not suppress the finding.

### RH-AWE-004 — Memory write without review

**Default severity:** medium

Trigger when:

- memory is enabled;
- writes are allowed;
- review is not required.

Severity becomes high when deployment stage is `production` and memory retention is `indefinite`.

### RH-AWE-005 — Insufficient tool-call logging

**Default severity:** medium

Trigger when:

- the workflow declares at least one write-capable, destructive, or code-execution tool;
- and `monitoring.log_tool_calls` is false;
- or the relevant tool does not declare adequate argument logging.

The first implementation may emit separate findings for global and tool-level gaps.

### RH-AWE-006 — Code execution outside a toy workflow

**Default severity:** high

Trigger when:

- a tool has `mode: code_execution`;
- and deployment stage is `internal` or `production`.

The rule may be accepted by a documented exception but remains in the report.

### RH-AWE-007 — Sensitive data reaches a less trusted output boundary

**Default severity:** critical for `restricted`, high for `confidential`

Trigger when:

- a flow carries `confidential` or `restricted` data;
- and the destination output permits only a lower sensitivity;
- or the destination boundary is `public`;
- or the destination is external without declared approval.

The evaluator compares classifications using:

```text
public < internal < confidential < restricted
```

### RH-AWE-008 — Weak or unknown execution identity

**Default severity:** low for read-only tools, medium otherwise

Trigger when:

- a tool identity is `unknown` or `anonymous`;
- and the tool is not a purely synthetic toy read tool.

Severity becomes high for destructive or code-execution tools.

## 9. Finding model

Every finding must contain:

```json
{
  "rule_id": "RH-AWE-001",
  "title": "External write without approval",
  "severity": "high",
  "status": "open",
  "object_ref": "tool:send_customer_email",
  "yaml_paths": [
    "tools[1].mode",
    "tools[1].requires_approval"
  ],
  "evidence": {
    "mode": "external_write",
    "requires_approval": false
  },
  "why_it_matters": "External writes can publish manipulated or private content.",
  "recommendation": "Require approval and restrict allowed recipients or domains."
}
```

Allowed severities:

```text
info | low | medium | high | critical
```

Allowed statuses:

```text
open | accepted_exception
```

## 10. Overall result

Version `v0.1` does not calculate a numerical security score.

The report status is:

- `pass`: no open findings at or above the configured blocking threshold;
- `review_required`: at least one open finding below the blocking threshold;
- `fail`: at least one open finding at or above the blocking threshold;
- `invalid`: schema or referential validation failed.

Default blocking threshold: `high`.

Severity order:

```text
info < low < medium < high < critical
```

Accepted exceptions remain visible but do not contribute to the blocking result unless expired.

## 11. Policy configuration

An optional policy YAML controls thresholds and enabled rules.

```yaml
policy_version: "0.1"
blocking_severity: high
enabled_rules:
  - RH-AWE-001
  - RH-AWE-002
  - RH-AWE-003
  - RH-AWE-004
  - RH-AWE-005
  - RH-AWE-006
  - RH-AWE-007
  - RH-AWE-008
```

Requirements:

- Unknown rule IDs are rejected.
- Rules are enabled by default when no policy is supplied.
- Policy cannot disable schema validation.
- Policy configuration must be recorded in the report metadata.

## 12. CLI

Package command:

```bash
rackhead-eval
```

Required commands:

```bash
rackhead-eval workflow.yaml
rackhead-eval workflow.yaml --format markdown
rackhead-eval workflow.yaml --format json
rackhead-eval workflow.yaml --policy policy.yaml
rackhead-eval workflow.yaml --output report.md
rackhead-eval workflow.yaml --blocking-severity medium
```

### Exit codes

- `0`: valid workflow and no blocking open findings;
- `1`: valid workflow with blocking open findings;
- `2`: invalid workflow, invalid policy, or evaluator configuration error;
- `3`: unexpected internal evaluator failure.

Human-readable errors go to stderr. Reports go to stdout unless `--output` is provided.

## 13. Markdown report

Minimum structure:

```markdown
# Agent Workflow Security Review

## Summary

- Workflow
- Workflow version
- Schema version
- Evaluator version
- Deployment stage
- Result
- Blocking threshold
- Finding counts by severity
- Accepted exception count

## Findings

### RH-AWE-001: External write without approval

- Severity
- Status
- Object
- YAML paths
- Evidence
- Why it matters
- Recommendation

## Accepted exceptions

## Validation notes

## Limitations
```

The limitations section is mandatory and states that the report evaluates declared configuration only.

## 17. Testing requirements

### Unit tests

- each rule has positive and negative tests;
- severity adjustments are tested;
- exceptions are tested;
- severity ordering is tested;
- report status calculation is tested.

### Validation tests

- structural errors;
- referential errors;
- duplicate IDs;
- invalid memory retention;
- unknown rule IDs;
- expired exceptions.

### Integration tests

- CLI Markdown output;
- CLI JSON output;
- output file behavior;
- exit codes `0`, `1`, and `2`;
- default policy and custom policy.

### Snapshot tests

- stable Markdown report for the risky support agent;
- stable JSON report for the same fixture.

## 18. CI requirements

Every pull request must run:

```text
ruff check
ruff format --check
mypy
pytest
```

CI must not require secrets or network access for the test suite.

## 19. Security and privacy requirements

- No production credentials or personal data in fixtures.
- No outbound network calls in core evaluation.
- No dynamic code execution.
- YAML parsing must use safe loading.
- Error messages must not print unrelated environment variables or secrets.
- Reports must include only values present in the supplied workflow or evaluator-generated metadata.
- Dependency versions should be reviewed and pinned through the project packaging strategy.
- A security policy remains available at repository level.

## 20. Compatibility and versioning

Three versions are tracked separately:

- evaluator package version;
- workflow schema version;
- report format version.

`v0.1.0` supports workflow schema `0.1` and report format `0.1`.

Breaking schema changes require a new schema version. Rule behavior changes that materially affect findings must be documented in release notes.

## 22. Follow-up work after v0.1

Potential later work, explicitly excluded from the first release:

- SARIF output;
- GitHub Action wrapper;
- framework-specific importers;
- runtime trace ingestion;
- richer policy-as-code;
- threat-framework mappings;
- graph visualization;
- taint-style propagation;
